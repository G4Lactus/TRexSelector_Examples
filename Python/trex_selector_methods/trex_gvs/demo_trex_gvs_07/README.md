# Demo 07: T-Rex+GVS on Heterogeneous ARMA Blocks (Python)

## Purpose

Stress-test T-Rex+GVS when different blocks follow **different time-series
structures** rather than a single uniform correlation model. This is the most
heterogeneous of the four-block scenarios: each active block has its own ARMA
specification, testing whether the selector can still recover grouped signal when
"the group correlation model" is not one fixed thing.

Python port of `cpp/.../demo_trex_gvs_07_mc_sim_arma_blocks`. The R counterpart is
`R/trex_selector_methods/trex_gvs/demo_trex_gvs_07/`.

Indices are **0-based** throughout (Python convention); the R counterpart is
1-based, so block IDs and column ranges here are shifted down by one relative to
that README.

## Data-generating process

`dgp_arma_blocks_snr` (from the shared `trex_gvs_dgps` module) ‚Äî four blocks of
sizes `{20, 50, 80, 65}`, each generated by a distinct time-series process,
shuffled into `p = 500` columns with white-noise gaps and standardised to unit
column variance. Each series is produced by the `_sim_arma` helper, a direct ARMA
recursion with burn-in `x_t = eps_t + sum_k ar[k] x_{t-k} + sum_k ma[k] eps_{t-k}`:

- **Block 0** (20 vars, active): AR(1), coefficient `ar_coef`.
- **Block 1** (50 vars, active): MA(3), `ma_coefs = (0.5, 0.3, 0.1)` (fixed).
- **Block 2** (80 vars, active): ARMA(2,1), `ar = (ar_coef, ar_coef/5)`,
  `ma = (ma_coefs[0],)`.
- **Block 3** (65 vars, inactive trap): AR(1), coefficient `ar_coef`.

The three active blocks carry `beta = 3` (`s = 150`). The swept correlation knob
is `ar_coef` (shared by the AR/ARMA blocks) ‚Äî **not** `rho`, since the MA(3) block
has no AR component. Dimensions: `n = 200`, `p = 500`.

Because the ARMA series are built with a pure-Python per-row recursion rather than
a vectorised backend, data generation is comparatively slow ‚Äî a single Part-1 run
takes roughly 20 seconds.

## Methods compared

- **EN** ‚Äî `gvs_type = "EN"`, `en_solver = "TENET"`
- **EN+AUG** ‚Äî `gvs_type = "EN"`, `en_solver = "TENET_AUG"`
- **IEN** ‚Äî `gvs_type = "IEN"`, `en_solver = "TENET"`

Shared controls (from `CFG`): `K = 20`, `tFDR = 0.1`, `corr_max = 0.98`,
`hc_dist = "single"`, `num_MC = 200`, `seed = 2026`, and for the sweeps
`ar_coef = 0.8` (fixed for the SNR sweep) / `snr = 2.0` (fixed for the ar_coef
sweep).

## Parts

- **Part 1** ‚Äî Single-run demo: block layout (with per-block model types and
  0-based column ranges), an MA(3) lag-structure check (Block 1 lag-1 correlation
  expected `> 0`, lag-4 expected `~ 0`), then one EN / EN+AUG / IEN run each.
- **Part 2** ‚Äî SNR sweep at fixed `ar_coef = 0.8`; `snr in {0.1, 0.2, 0.5, 1.0,
  2.0, 5.0}`.
- **Part 3** ‚Äî `ar_coef` sweep at fixed `snr = 2.0`; `ar_coef in {0.10, 0.20,
  0.30, 0.40, 0.50, 0.60, 0.70, 0.80, 0.90, 0.95}`.
- **Part 4** ‚Äî 2-D `snr x ar_coef` grid (5 SNR `{0.2, 0.5, 1.0, 2.0, 5.0}` x
  6 ar_coef `{0.30, 0.50, 0.70, 0.80, 0.90, 0.95}`), printing `mean_TPP` and
  `mean_FDP` matrices per method.

## Running

Only **Part 1 runs by default**; the heavy Monte Carlo parts (2-4) are opt-in via
the `RUN_PART_2` / `RUN_PART_3` / `RUN_PART_4` flags at the top of the file. (This
differs from the R suite, where all parts default on.)

```bash
.venv/bin/python Python/trex_selector_methods/trex_gvs/demo_trex_gvs_07/demo_trex_gvs_07.py      # Part 1 only (default)
.venv/bin/python Python/trex_selector_methods/trex_gvs/demo_trex_gvs_07/demo_trex_gvs_07.py 8    # 8 worker cores
```

The optional positional argument sets the number of Monte Carlo worker cores
(default `min(6, os.cpu_count())`). The file inserts both its own directory and the
parent suite directory onto `sys.path`, so the shared `trex_gvs_dgps` and
`gvs_sim_common` modules resolve and remain re-importable by the `spawn`-launched
workers ‚Äî run the demo as a file, not via `python -`. Output is console-only (no
result files, no Plotly heatmap is rendered).

## Interpretation

- Because the three active blocks have genuinely different correlation structures
  (AR(1), MA(3), ARMA(2,1)), recovery is expected to be uneven across blocks ‚Äî
  aggregate TPP is a blend of these different per-block difficulties.
- FDP should stay reasonably controlled near or below `tFDR = 0.1`; the trap block
  (AR(1), same `ar_coef` as the active AR(1) block) is structurally similar to an
  active block and is the main test of false-group exclusion.
- Compare against Demo 06 (uniform AR(1) blocks) to see whether cross-block
  heterogeneity costs additional power beyond what AR(1) decay alone costs.
